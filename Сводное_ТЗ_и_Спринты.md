## 1. Введение и цели

Документ объединяет финальное ТЗ v1.0, ТЗ v3.0, анализ пробелов и дизайн-макеты в единый план реализации. Назначение — дать команде разработки чёткий пошаговый маршрут от первого коммита до релизной сборки. Все спринты выверены так, чтобы каждый следующий опирался на готовую базу предыдущего и закрывал максимум реальных бизнес-сценариев из ТЗ заказчика.

### 1.1 Что мы строим

Мобильное приложение iOS/Android для управления ремонтными и строительными проектами. Приложение даёт прозрачность по срокам, финансам и качеству работ всем участникам — заказчику, бригадиру, мастеру, представителю — без лишних звонков и сообщений.

### 1.2 Ключевые цифры проекта

| Параметр | Значение |
|---|---|
| Общий срок разработки | 18 рабочих дней |
| Бекенд-спринты | 5 спринтов (10 дней) |
| Flutter-спринты | 4 спринта (8 дней) |
| Длина спринта | 2–3 дня |
| Уникальных экранов в дизайне | ~180 (6 кластеров) |
| Системных ролей | 4 (заказчик / представитель / бригадир / мастер) |
| Доменных модулей | 16 (auth, project, stage, step, approval, finance, materials, tool, chat, doc, feed, push, notes, methodology, template, admin) |
| Платформы | iOS 14+, Android 7.0+ (API 24) |

### 1.3 Дисциплина спринтов

- Спринт всегда заканчивается работающим инкрементом (можно продемонстрировать).
- Каждая задача спринта связана с конкретным разделом исходного ТЗ и/или экраном дизайна.
- В конце каждого спринта — короткое демо, прогон чек-листа Definition of Done, обновление API-документации (OpenAPI).
- Бекенд и мобильное приложение разрабатываются строго последовательно: к началу мобильных спринтов API стабилизирован.
- Контракт API фиксируется в OpenAPI-схеме сразу после первого спринта — это позволит мобильной команде начать с генерации DTO/клиентов.

## 3. Технологический стек и архитектура

Стек подобран под параметры: высокий темп разработки (18 дней), сложная доменная логика, real-time чаты, push-уведомления, файлы с долгим хранением, гибкая RBAC-матрица, потенциал к росту нагрузки и миграции на масштаб.

### 3.3 Архитектура высокого уровня

Бекенд — модульный монолит на NestJS с чёткой границей доменов (DDD-light). Это упрощает развёртывание на старте и оставляет дверь открытой для выноса доменов в микросервисы (chat, files) при росте нагрузки.

- Domain modules: auth, users, roles, projects, stages, steps, approvals, materials, tools, finance, chat, documents, feed, notifications, templates, methodology, admin.
- Поперечные слои: rbac (политики прав), audit (event log в ленту), files (S3-обёртка), realtime (gateway), notifications (FCM + email future).
- Лента событий — централизованная outbox-таблица; все доменные изменения публикуют события, лента подписана и пишет читаемые записи.
- RBAC — комбинация (роль_в_системе × участие_в_сущности × права_представителя). Решение в отдельном сервисе AccessGuard, оборачивающий каждую API-команду.
- Flutter — Clean Architecture: presentation (виджеты) / application (riverpod-провайдеры, usecases) / domain (модели + интерфейсы) / data (DTO, dio-клиенты, drift-репозитории).

## 5. Что мы дополнительно учли (вне явного ТЗ)

Финальное ТЗ закрывает большинство граничных случаев из gaps-анализа. Ниже — пункты, которые мы добавили инициативно: они не противоречат заказчику, но критичны для прод-стабильности.

### 5.1 Безопасность и приватность

- Хранение паролей: bcrypt (cost 12). Никаких plain/MD5.
- Refresh-токены ротируются и привязаны к устройству (deviceId + ip-fingerprint).
- Rate limiting: 5 попыток входа / 5 минут / IP+phone. После — блок 5 минут.
- **Idempotency-Key** для финансовых операций (createPayment, confirmPayment, createSelfPurchase) — защита от дубля при потере связи.
- Подписанные URL для скачивания файлов (TTL 5 минут), запрет прямого доступа к S3.
- Защита от энумерации телефонов: единый ответ при «номер не зарегистрирован/неверный пароль» в восстановлении.

### 5.2 Производительность и UX

- Светофор и прогресс — материализованные представления: пересчёт по триггеру при изменении шага/паузы/дедлайна, не при каждом GET.
- Фото — компрессия на клиенте до 1920px / 80% качества + EXIF-zero (защита геолокации).
- Превью документов: PDF — первая страница как PNG (генерация в очереди); офисные форматы — иконка + название + размер.
- Курсорная пагинация (cursor based) для лент, чатов, согласований — стабильнее offset при активной записи.
- ETag / If-None-Match на «справочные» запросы (методичка, шаблоны, FAQ) для оффлайн-кеша.

### 5.3 Real-time и оффлайн

- WebSocket-комнаты: `project:{id}`, `stage:{id}`, `chat:{id}`, `user:{id}` — точная фильтрация рассылки.
- Heartbeat 25 сек, переподключение с экспоненциальной задержкой (1s → 30s).
- Очередь отложенных действий в drift — без сети можно отметить шаг, прикрепить фото, оставить заметку; при восстановлении сети — синхронизация (с предупреждением о конфликтах).

### 5.4 Долгое хранение (10+ лет)

- Архивные проекты переносятся в S3 storage class STANDARD_IA / COLD после 1 года неактивности.
- ZIP-выгрузка архива проекта (фото + документы + лента в PDF) — фоновая задача, ссылка приходит push'ом.
- DB-архив: партиционирование `feed_events` и `chat_messages` по году.

### 5.5 Наблюдаемость и поддержка

- Sentry для бекенда и Flutter — single-source ошибок.
- Бизнес-метрики в Grafana: количество активных проектов, среднее время цикла этапа, % просрочек, доля споров по выплатам.
- Health-check'и для очередей и FCM — вебхук в Telegram-чат разработчиков.

### 5.6 Юридические и продуктовые мелочи

- Согласие на обработку персональных данных при регистрации (152-ФЗ) — обязательный чекбокс, ссылка на политику.
- Версионирование политики и пользовательского соглашения — пересогласие при смене версии.
- Формат сумм — единый Money value-object на бекенде (minor units / int64), форматирование на клиенте.
- Тайм-зоны — все timestamp в UTC; на клиенте конвертация в локальную.
- Локализация дат: всегда ДД.ММ.ГГГГ, время по локали устройства.

### 5.7 Подготовка к Pro-версии и платным фичам (заложить, не реализовывать)

- Сущность Subscription / FeatureFlag заложена в схему БД — фичи помечены тегом `plan: free/pro`.
- Точка интеграции с эквайрингом (CloudPayments / ЮKassa) обозначена как stub-сервис.

# Backend Foundation: каркас, Auth, Users, Roles, RBAC

#### Цель

Подготовить инфраструктуру и базовые сервисы, на которых вырастут все доменные модули. К концу спринта приложение умеет: регистрировать пользователя по телефону, логинить, восстанавливать пароль через SMS, переключать активную роль, отдавать профиль и список ролей.

#### Задачи по дням

**День 1 — Инфраструктура**

- Репозиторий, NestJS-monorepo (apps/api, libs/*), Prettier + ESLint + Husky.
- Docker Compose: api, postgres 16, redis 7, minio.
- Конфиг через @nestjs/config + Joi-валидация .env.
- Prisma init, базовая схема, первая миграция.
- OpenAPI (Swagger UI) на `/api/docs`, заглушка эндпоинта `/healthz`.
- CI: lint, build, миграции, тесты — на каждый PR.

**День 2 — Auth + Users + Roles + RBAC + Files**

- Модели: User, UserRole, DeviceToken, Session.
- `POST /auth/register` (phone+pass+name+role) — без SMS, как в финале §1.1.
- `POST /auth/login` (phone+pass) → access (15 мин) + refresh (30 дн).
- `POST /auth/refresh`, `POST /auth/logout`.
- Восстановление пароля через SMS-провайдера: `/auth/recovery/send`, `/verify`, `/reset` (3 попытки → блок 5 мин).
- `GET /me`, `PATCH /me` (имя, фамилия, аватар, язык).
- `POST /me/roles`, `DELETE /me/roles/:role` — добавить/убрать роль (без перелогина).
- `PUT /me/active-role` — переключить активную роль.
- `POST /me/devices` — регистрация FCM-токена.
- Сервис **AccessGuard** + декоратор `@RequireAccess('action', resourceLoader)` + базовая матрица прав 4 ролей × 16 действий из финала §1.5.
- Files-модуль: presigned upload в MinIO, presigned download, ограничение по mime/size, обёртка sharp для thumbnails.

#### Артефакты

- Деплоится `docker compose up` — БД мигрирует, можно зарегистрироваться через Postman.
- OpenAPI v0.1: 12 эндпоинтов auth + me.
- Документ `ARCHITECTURE.md` (структура папок, поток запроса, RBAC).

#### Definition of Done

- Покрытие unit-тестами AuthService и AccessGuard ≥ 70%.
- e2e-тест happy-path: register → login → /me → switch role.
- Линтер 0 ошибок, миграции применяются на чистой БД.
